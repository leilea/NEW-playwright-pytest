## 上下文

当前回放失败时，错误信息通过 `_parse_playback_error()` 解析，其中定位器提取依赖 `_extract_locator_from_context()` 做**回溯文本搜索**（从 Playwright 错误位置向前搜索 800 字符内的 locator 调用）。这种方式不可靠：

- 多步骤脚本中可能匹配到错误步骤之前的 locator 调用
- 使用 `_safe()` 封装时，内部策略数组格式对用户不友好
- 无法关联到失败步骤的序号
- 搜索窗口 800 字符对长脚本不够

已有 spec `playback-error-prompt` 要求"显示失败定位器和对应步骤序号"，实现未满足。

## 目标 / 非目标

**目标：**
- 回放失败时精确标识失败的是第几步、什么操作、什么定位器
- 生成脚本中植入步骤标记，解析时通过标记定位，不依赖回溯搜索
- 前端 error-card 显示步骤序号和操作类型
- 向后兼容：旧脚本仍可通过回溯搜索得到部分信息

**非目标：**
- 不修改 `_extract_locator_from_context()` 函数（保留作为回退）
- 不修改 WebSocket 协议格式
- 不修改定位器友好展示格式（保持原始策略列表）
- 不做实时进度推送（方案 C）

## 决策

### 决策 1：使用 stdout 打印标记代替 IPC 通信

**选择**：在生成的 pytest 脚本中每步前插入 `print(f"__STEP_MARKER__:{i}")`，`i` 为步骤序号。

**替代方案**：
- **WebSocket 实时进度**：每步开始时通过 WS 推送当前步骤 — 改动大，需修改子进程通信架构
- **临时文件**：每步写入步骤 ID 到文件 — 文件 IO 有竞态风险
- **stderr 标记**：写入 stderr — 但 stderr 内容更多是错误输出，标记夹杂其中不易解析

**理由**：stdout 打印是最轻量的方式，pytest `-s` 模式下直接输出到 subprocess stdout pipe，解析可靠。

### 决策 2：标记仅包含步骤序号，定位器信息从原始 steps 数据查表

**选择**：标记只包含 `{i}`（步骤索引）。解析时通过索引在 `steps` 列表中查找 `action` 和 `selector`。

**替代方案**：
- 标记中同时包含 `action:selector` 字符串 — 需要处理 selector 转义问题
- 标记包含 JSON 序列化 — 增加解析复杂度

**理由**：`steps` 列表在 `run_playback()` 中原样可用，传递给 `_parse_playback_error()` 即可查表。标记最小化，避免序列化/转义问题。

### 决策 3：标记放在步骤代码之前，而非之后

**选择**：`print` 在 handler 代码之前生成。

**理由**：标记在步骤执行前打印。如果步骤执行时出错，错误位置总是在标记之后，保证 `__STEP_MARKER__` 在错误发生前已输出到 stdout。

### 决策 4：`_parse_playback_error()` 接受 `steps` 参数

**选择**：签名改为 `_parse_playback_error(stdout, stderr, steps=None)`。

**替代方案**：在 `run_playback()` 中自行解析标记 — 导致解析逻辑分散。

**理由**：错误解析逻辑集中在一个函数中，保持单一职责。

## 风险 / 权衡

- **[低风险] 生成脚本修改**：现有脚本是临时的（每次回放生成新文件），旧逻辑作为回退不受影响
- **[低风险] 多步骤中标记位置**：导航型 click 前有 `with expect_navigation` 包裹，标记插在 `with` 代码块外，确保标记在预期导航超时时也能被识别
- **[无风险] 适配器模式**：所有改动向后兼容 — 旧运行记录没有标记，回退到 `_extract_locator_from_context()`
