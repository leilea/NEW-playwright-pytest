## 需求
### 需求:套件以「系统名称 + 版本号」组合唯一标识

系统必须保证同一「系统名称 + 版本号」组合下不能存在两个套件；创建 / 修改套件时若冲突必须阻止写入并向用户报错。

#### 场景:创建重复套件被拒绝
- **当** 用户在「新增套件」对话框中输入「电商后台 / v1.0」并提交
- **并且** 系统中已存在同名同版本套件
- **那么** 系统必须拒绝创建并显示「该系统名称 + 版本号的套件已存在」错误，不修改持久化文件

#### 场景:创建唯一套件成功
- **当** 用户提交「电商后台 / v1.0」且系统中无重复
- **那么** 系统必须创建一条新套件记录，`id` 为 uuid4，`created_at` / `updated_at` 为当前 ISO 时间，`created_by` 为当前登录用户名

### 需求:套件列表展示核心元数据

打开「测试用例」页面后，系统必须**默认显示套件列表**（不是用例列表），表格至少包含以下列：
- 序号（行号，从 1 开始）
- 系统名称（可点击进入）
- 版本号
- 用例数量（动态统计）
- 创建人
- 上次操作时间（`updated_at`，格式 `YYYY-MM-DD HH:MM`）
- 操作（进入 / 修改 / 删除 三个按钮）

#### 场景:打开测试用例页默认进入套件列表
- **当** 用户在侧边栏点击「测试用例」菜单
- **那么** 系统必须渲染套件列表视图，不渲染用例列表

#### 场景:列表顶部提供新增按钮
- **当** 套件列表渲染
- **那么** 系统必须在列表上方显示「+ 新增」按钮，点击后弹出「新增套件」对话框

#### 场景:用例数量动态统计
- **当** 套件列表渲染
- **那么** 系统必须对每行套件计算「该套件下 `suite_id == suite.id` 的用例数」并展示，0 时显示「0」

### 需求:系统名称可点击进入该套件下的用例列表

套件列表中「系统名称」必须渲染为可点击元素；点击系统名称或操作栏「进入」按钮都进入该套件下的用例列表（即当前的用例列表页）。

#### 场景:系统名称点击进入
- **当** 用户点击套件列表某行的「系统名称」可点击元素
- **那么** 系统必须把 `tc_view` 切到 `case_list`，并把 `tc_active_suite_id` 设为该套件 id，渲染该套件下的用例列表

#### 场景:操作栏进入按钮
- **当** 用户点击操作栏「进入」按钮
- **那么** 系统行为与"点击系统名称"完全一致

### 需求:操作栏可修改套件

操作栏「修改」按钮必须弹出对话框，允许编辑「系统名称」与「版本号」，保存后立即刷新列表中该行的数据；组合唯一约束在修改时同样生效。

#### 场景:打开修改对话框
- **当** 用户点击某行的「修改」按钮
- **那么** 系统必须弹出「修改套件」对话框，表单预填当前「系统名称」「版本号」

#### 场景:修改冲突被拒绝
- **当** 用户在修改对话框中把「系统名称 / 版本号」改为与另一已存在套件冲突
- **那么** 系统必须拒绝保存并显示冲突错误，该套件数据不变

#### 场景:修改成功刷新列表
- **当** 用户修改成功保存
- **那么** 套件 `updated_at` 必须刷新为当前时间，列表中该行数据立即更新

### 需求:非空套件禁止删除

套件下用例数 > 0 时，操作栏「删除」按钮必须置灰且不可点击；用例数 == 0 时可点击。删除前必须有二次确认。

#### 场景:非空套件删除按钮置灰
- **当** 某套件下 `case_count > 0`
- **那么** 该行操作栏「删除」按钮必须 `disabled=True`，鼠标悬停提示「套件内还有 N 条用例，请先删除」

#### 场景:空套件可删除且有二次确认
- **当** 某套件下 `case_count == 0` 且用户点击「删除」按钮
- **那么** 系统必须弹出二次确认；用户确认后从 `suites.json` 移除该条记录并刷新列表

#### 场景:后端拒绝绕过 UI 的非空删除
- **当** 服务层 `delete_suite_if_empty(suite_id)` 被调用且 `case_count > 0`
- **那么** 系统必须返回 `(False, "套件内还有 N 条用例，请先删除")`，不修改 `suites.json`

### 需求:套件下用例增删改自动刷新套件上次操作时间

当某套件下的用例被新增 / 修改 / 删除时，该套件的 `updated_at` 必须自动刷新为当前时间。

#### 场景:新增用例刷新套件时间
- **当** 用户在套件 A 下新增一条用例成功
- **那么** 套件 A 的 `updated_at` 必须被写为当前 ISO 时间，套件列表中该行的「上次操作时间」列立即变化

#### 场景:修改用例刷新套件时间
- **当** 用户在套件 A 下修改一条用例成功
- **那么** 套件 A 的 `updated_at` 必须被刷新

#### 场景:删除用例刷新套件时间
- **当** 用户在套件 A 下删除一条用例成功
- **那么** 套件 A 的 `updated_at` 必须被刷新

### 需求:启动期把无主用例自动归入「默认系统」

首次部署或历史 `testcases.json` 中存在无 `suite_id` 字段的条目时，系统必须启动期一次性迁移到「默认系统 / v1.0」套件下，且该迁移**幂等**。

#### 场景:首次启动创建默认套件
- **当** 启动时 `suites.json` 中无 `name == "默认系统"` 套件
- **并且** `testcases.json` 中存在无 `suite_id` 的用例
- **那么** 系统必须创建一个 `name="默认系统"`、`version="v1.0"`、`created_by=当前登录用户` 的套件，并把所有孤儿用例的 `suite_id` 写为该套件 id

#### 场景:二次启动跳过迁移
- **当** 启动时所有用例均已有 `suite_id`
- **那么** 系统必须跳过迁移逻辑，不修改任何文件

#### 场景:迁移幂等
- **当** 启动两次且中间没有数据写入
- **那么** 第二次启动必须检测不到孤儿，迁移逻辑不重复创建「默认系统」

### 需求:用例列表必须按当前套件过滤

用例列表页（`tc_view == "case_list"`）必须仅展示 `tc_active_suite_id` 下的用例，且页面顶部提供面包屑指明所属套件 + 「返回套件列表」入口。

#### 场景:用例列表过滤
- **当** 用户在套件 A 下进入用例列表
- **那么** 系统必须仅渲染 `suite_id == A.id` 的用例；其他套件下的用例不出现

#### 场景:用例列表面包屑
- **当** 用例列表页渲染
- **那么** 系统必须在顶部显示 `套件 / 系统名称 版本号` 面包屑，并提供「← 返回套件列表」按钮，点击后回到套件列表视图

### 需求:创建人取自当前登录用户名

套件的 `created_by` 字段必须从系统当前登录用户名读取，读不到时兜底为 `"admin"`。

#### 场景:有环境变量时取环境变量
- **当** 系统 `USER` 或 `USERNAME` 环境变量非空
- **那么** 系统必须把该值作为 `created_by` 写入新建套件

#### 场景:无环境变量时兜底
- **当** 系统 `USER` 与 `USERNAME` 环境变量均未设置
- **那么** 系统必须把 `created_by` 写为 `"admin"`
